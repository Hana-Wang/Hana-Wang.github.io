Database concepts-1

—-What is Database? What is a language used to interact with Database?

  • Database is a way of storing and organising data electronically.

  • Relational database is a database where data is organised into tables where related data is stored in rows and columns. Relationships can be defined between tables inside the database. It has some similarity with spreadsheet applications which also organise data in the forms of rows and columns.

  • However, database can be more poweful. Database can store much more data. Once the data is stored inside the database, many users can write queries together to gather insights from the database at the same time. In addition, database can be secured with encryption.

  • One most common language, used for creating, querying and updating relational databases, is called SQL, structured query language.

—-Understanding how data is organised inside a database

—-What are tables inside a database?

  • The main building block of databases are tables; Database is organised into tables, which hold related data about a particular subject/entity.

  • Tables are organised into rows and columns. Rows are often referred to as records. Columns are referred to as fields.

  • A table’s fields are limited to a specific set when the database was created. But the number of rows is unlimited.

—-Table Naming

  • lowercase, not include spaces. Use underscores in place of spaces.

  • A table name refer to a collective group. Like inventory, but it is also okay for the table to have a plural name such as “products”

—-Record

  • A record is a row in a table; it holds data on an individual entity observation
  • For example, a ‘products’ table has a number of records, each record for each product. The record for product A, indicates the pieces of information about the product

—-Field

  • A field is a column in a table; it holds one piece of information about all observations in the table.
    For example, the ‘brand’ field in the ‘products’ table lists all of the brands of our library products.

Notice that

  • field name should be singular, rather than plural. Because it refers to one piece of information contained in that field for a single record. For example, use product_name fields rather than product_names
  • Two fields in a table cannot have the same name, because each field represent one piece of information about all observations in the table.
  • Field name should not share the same name with the table names. It represents one piece of the information about the table entity.

One special field is unique identifier, which is called a key, and its value can distinguish different records in the same table. This value is very often a number.

The most suitable choice for a unique identifier is
a field that

  • Its value is a number that is unique across all entity observations/records,
  • we can expect that new observation/record will receive the unique field value as well

For example, customer_id can be a field for customers table.

  • The field value is unique across all current customers,

  • and we can expect that new customers will receive unique customer_ids as well.

When creating tables in a database, it is better to organise data into more separate subject tables rather than few tables that combines information about multiple subjects. Table subject/topics should remain separate.

For example, In a library database, we can have ‘cutomers’ table and ‘checkouts’ table separately, rather than putting all them together as a table ‘customer_checkouts’. This is because one customer can have none or multiple checkouts. Although the data are the same, but if organised the combined information into a table, that can lead to record with missing fields values(one customer can have none checkouts) or duplicate information where ‘customer_num’ that distinguish customer is no longer unique (one customer can have multiple checkouts).

For example, customers table contains fields: card_num, name, member_year, total_fine.
checkouts table contains fields: checkout_id, start_date, due_date, book_id, card_num

However, customer_checkouts table contain fields: card_num, name, member_year, total_fine, checkout_id, start_date, due_date, book_id

After data is stored inside a database, we can familiarize tables inside database, and then use SQL to gather information from multiple related tables according the query needs.

Learning Notes-2-AWS Data Engineer Associate

– Understanding Streaming Ingestion and Batch Ingestion

—What is Streaming Ingestion?

  • Streaming Ingestion is a method of continuously collecting and processing data as it is generated in real-time;

—Why we need real-time monitoring or analytics?

  • For example, temperature data can be recorded in a sensor and generated constantly, such as every second. Social media feeds can be generated constantly etc.
  • Sometimes, fraud data/information need to be detected in some context such as financial transaction data, when the data arrives, we need to have the immediate detection and corresponding reaction. This is where the real-time monitoring or real-time analytics comes in.

–AWS tools for Streaming Ingestion?

—-Amazon Web Service(AWS) provides relevant tools for real-time data ingestion (Streaming Ingestion).

  • Amazon Kinesis Data Streams can be used to ingest/collect the data and process the real-time data at scale. For example, temperature data generated in real-time can be sent to AWS Kinesis Data Streams using an AWS SDK such as boto3 from a Python script or from another application.

  • Next, the data can be processed immediately so that real-time temperature data changes can be monitored. For example, we can use serverless computing service AWS Lambda to connect Kinesis service as a event trigger, then the data can be read from the Kinesis Data Stream and processed in AWS Lambda; The processing functionality in python code can then be written inside the Lambda handler to process data as the data flows/arrives in Kinesis and trigger AWS Lambda.

—-Within an Amazon Kinesis Data Stream, how data is organised?

  • Data records can be added to Kinesis Data Stream
  • Each data record can go to a specific shard. Shard is a unit of parallel processing in Kinesis;
  • PartitionKey is a value associated with each data record you sent to the Kinesis Stream. PartitionKey is a user defined string that Kinesis uses to determine the shard location where each record will be stored.
  • The key is hashed by Kinesis using PartitionKey to decide which shard to assign the data record to. All data records with the same PartitonKey will go to the same shard (Partitionkey -> which shard the data goes to).

—-Why do we need a partition key?

  • It can help data distribution across shards. The under hood principle is hashing; It Applies a hash function to the partitionkey, the result of the hash function determines which shard the data goes to
  • Data can be divided into shards, which allow parallel processing of data records.
  • By specifying different partitionkeys, you can distribute records across multiple shards. Shards help distribute incoming data across multiple processing units; making it easier to handle high data loads;
  • This avoids overloading a single and allows the stream to handle more data simultaneously

—What is Batch Ingestion?

Batch ingestion collects data over a period of time and processes it in bulk at scheduled intervals. This is useful when data does not need to be processed immediately and can be handled together, reducing resource usage.

—-Why Use Batch Ingestion?

  • Scheduled Processing: Ideal for generating daily or weekly summaries and reports.
  • Efficient for Large Data Sets: Processes data all at once instead of continuously, saving resources.

—-AWS Tools for Batch Ingestion?
For example, we can use

  • Amazon S3: Stores incoming batch data files.
  • AWS Lambda: Triggers when new data files are added to S3, then processes each file in bulk.

—-Example Use Case with AWS Lambda:

Suppose you receive daily sales data from multiple stores. Each day, a CSV file with sales data is uploaded to an S3 bucket. We can use AWS Lambda to process this file when it arrives:

  1. Data Collection in S3: Every day, a new CSV file is uploaded to an Amazon S3 bucket (e.g., daily_sales_data/sales_YYYYMMDD.csv).

  2. Lambda Trigger: Set up S3 Event Notifications to trigger an AWS Lambda function whenever a new file is uploaded. This function will read and process the file.

  3. Processing Data in Lambda:
    ○ The Lambda function reads the file content, processes it (e.g., calculating total sales), and then stores the processed results in another S3 location or a database for further analysis.

Learning Notes-1-AWS Data Engineer Associate

– Create a AWS account and use it

  • Go to AWS website, start by setting up a root user email address, which is primary email address associate with my AWS account.
    After signing up AWS Free Trial for 12 months, we can log in using ‘root user’ to AWS managment console.

  • In default situation, AWS allows user to operate without an explicit AWS account name. In this case, AWS may use your root user email as a placeholder or leave the account name blank in some views.

  • Whether the AWS account name is set up, it can be checked under ‘account name’ inside the account setting from the aws management console.

  • AWS does not allow multiple account names for the same account. But there is a way to use AWS Organisation service to manage and group multiple AWS accounts under a single billing entity.

– AWS S3 (simple storage service)

  • Storage Structure: Bucket(container for storage), Objects(files)

    • objects(such as actual files) can be uploaded to the S3 storage
    • object is stored in the bucket

  • Each bucket is created in a given region, because it means where the data is stored physically.

    • For latency reasons, we normally choose to put the files close to where it is accessed from. For example, where the data or application is accessed from, then choose the relevant given region for Bucket.
    • Regulatory Compliance in different regions can also be considered

Naming convensions of Bucket Name

  • Bucket name need to have globally unique name which is a unique name across all accounts and all regions.
  • Naming conventions:
    • Must begin and end with a letter or number
    • The length is between 3 min and 63 max characters
    • Only lowercase letter, numbers, dots(.), hyphens(-)
    • Not formatted as an IP address

How to identify object in a bucket?

- use Key

  • a key is what an object is identified within a Bucket. Every object is identified with a unique key, the key name is assigned by user.
  • the key can be assigned like file name directly, or the combination of a folder name and a file name, such as
    • example.txt
    • folder/example.txt
  • Bucket name is not part of the Key name.

Use Cases of AWS S3 Storage

  • It is used for backup and recovery
  • application such as website can be build based on S3 whenever we want to store something for applications
  • data archiving
  • Entire data lake can be built based on S3 buckets
  • Store data such as structured data, semi-structured data, files; then data can be used for analytical purpose, such as use Athena to query the data

Different storage classes for different Use Cases

  • Data Durability: the likelihood of loosing an object in a year, such as 99.999999999%(‘11 nines’) chance of losing one object in a year
  • Data Availability: the percentage of time that data can be accessed in a year (the data service is operational)
  • Data Lifecyle Rules: as the age of data (a file, a folder or an entire bucket) change, storage classes can be changed.
    • For example, in the first 30 days, we might want to access data frequently using Standard S3 storage class
    • After 30 days, data can be automatically archived stored with different storage class
    • use Lifecycle rules automatically defined the change of storage classes over time
  • Data Versioning: Previous versions of an object can be retreved when we make some accidental or deletion of the object

What is a data-driven organisation?

Organisation might not define themselves as a “data-driven organisation” or “non-data-driven organisation”, but to some extent they use data in their decision-making and operations in the modern world; they create stategies, goal and make decisions based on insights derived from data, as opposed to decisions based on the leaders’ opinions, experiences or “gut feelings”(intuition-based decisions).

For example, a data-backed decisions can be made in a retail company by analyzing real-time sales data, customer preferences, and market trends to identify the best popular products in each season, not only rely on the store manager’s experience regarding which product to stock in different seasons.

It is clear that data-driven decision can make organisations have a competitive edge because informed decisions can be made based on factual data insights. The extent to which organisations use data in their operations can reflect their culture. Some organisations might be in the process of becoming data-driven by modernizing their data infrastructure or shifting their culture.

What is not a data-driven organisation?

  • Decisions are made based on opinions, experiences or gut-feeling of leaders, professionals
  • There is no strong system/infrastructure for data collection, process, storage or analysis. Data is scattered, not easily accessible.
  • Data use is not integrated into daily operations and long-term planning, but occassionally used or used in a reactive way.
  • Data is not emphasized/valued or encouraged in backing up decisions across departments. This might because employees and leaders do not understand how to use it effectively.
  • Data can be collected, stored, but not properly organized, stored that make it difficult or complext to analyze to derive a meaningful insights that help organisations. Such as, What do you want to achieve with the data? How to organise data to make analysis accessible?

What is a data collection?

It is a process, where information can be gathered from different sources.
Data sources can be physical records, surveys, sensors, websites, customer interactions or any other sources that organisation can gather useful information.

For example, a coffeeshop might use the following data sources;

  • Physical Records/Digital Records; a logbook for keeping daily sales and the number of customers
  • Surveys/Receipt Surveys; customers can fill out a short survey to rate the service etc.
  • Website; a website that contains menu, allow placing order online. how people interact with website can be tracked such as number of visits, popular products etc.
  • Customer interactions; social media sentiment, store conversations

Better decisions can be made to help the coffee shop grow, such as adjusting the menu, improving customer services etc.
